# suggestions for IOC


Given what you already have in package.json, src/extension.js, and src/runtimeMonitor.js, the next step is to make the IOC itself a first-class object, not just the files and CA/PVA channels.

I’d structure it around 3 layers:

Static IOC model.
Parse st.cmd, dbLoadRecords, dbLoadTemplate, substitutions, macros, db/template/proto/dbd files, and build the resolved IOC configuration before it runs.

Managed IOC session.
Let the extension launch st.cmd with a saved profile: working directory, env vars, host architecture, optional pre-launch build step, and an embedded IOCSH console.

Runtime model.
Keep CA/PVA monitoring, but tie it back to the loaded IOC model and, when possible, IOCSH queries.

The highest-value features would be:

IOC launch profiles.
A launch.json-style config for IOC startup.
One-click Start IOC, Restart IOC, Stop IOC.
Persist stdout/stderr logs and parse startup failures.

IOCSH terminal integration.
A dedicated VS Code terminal or PTY for IOCSH.
Command palette wrappers for dbl, dbpr, dbpf, dbgf, epicsEnvShow, iocshCmd.
Click a record in the editor and run dbpr or dbpf against it.

Expected-vs-actual IOC diff.
From static parsing, know which records should exist.
From IOCSH dbl, know which records do exist.
Show missing records, duplicate loads, unresolved macros, failed load files, and records present at runtime but not in the model.

Startup/load graph view.
A tree like:
st.cmd -> dbLoadRecords/dbLoadTemplate -> substitutions -> db/template -> records -> links/proto
This would make IOC debugging much faster than jumping file to file.

Record inspector.
For one record, show:
Type, fields, alarms, scan, DTYP, INP/OUT, links, current value, and source file.
Then add safe actions like dbpf, process, or copy caget/caput command.

Log-aware diagnostics.
Parse IOC startup/runtime logs for common EPICS/asyn/StreamDevice patterns.
Link log errors back to file/line/record when possible.

Soft IOC / smoke-test workflow.
Run a temporary IOC locally from the workspace.
Validate that all db/template loads resolve.
Optionally snapshot dbl/selected field values for regression tests.

StreamDevice/asyn workflow.
Since many IOC problems are actually protocol/device issues, add:
proto-file navigation from INP/OUT,
protocol argument preview after macro expansion,
maybe a request/response log panel if you can tap the runtime.

If you want to make it feel like a real one-stop environment, I would build in this order:

Managed IOC launcher + IOCSH console
Static startup/load graph
Expected-vs-actual runtime diff
Record inspector with dbpr/dbpf
Log parsing and diagnostics
That sequence gives you an actual IOC workbench, not just a better syntax extension.

If you want, I can turn this into a concrete milestone plan with commands, views, data models, and the minimum architecture changes needed in this codebase.

## Purpose

This document describes how the current `vscode-epics` extension could evolve from a declarative editor-assistance package into a real EPICS language toolchain for VS Code.

Today, the extension is centered around:

- file-type registration in `package.json`
- TextMate grammars in `syntaxes/`
- snippets in `snippets/`
- basic editor behavior in `language-configuration.json`

That is enough for syntax coloring and snippet expansion, but it does not provide semantic understanding of EPICS source files. The next step is to treat EPICS artifacts as structured languages rather than plain text templates.

## Current State

The existing extension works well as a lightweight language-support package:

- It maps EPICS-related file extensions to VS Code language ids.
- It uses TextMate grammars to color tokens using regular expressions.
- It uses snippets to insert predefined templates for records, fields, and startup commands.

This model is useful, but it has hard limits:

- highlighting is lexical, not semantic
- snippets do not understand surrounding context
- there is no symbol table
- there is no project-wide cross-reference model
- there are no diagnostics, navigation, rename, or refactoring features

In short:

- TextMate grammar + snippets = editor convenience
- parser + symbol model + language server = real language tooling

## Goal

The goal is to make EPICS support feel like support for a real programming language in VS Code.

That means adding:

- semantic understanding of EPICS files
- workspace indexing
- symbol navigation
- validation and diagnostics
- context-aware completion
- refactoring and structural editing support

## What "Advanced" Means

The following features would move the extension beyond syntax highlighting and snippets.

### 1. Context-Aware Completion

Instead of offering static snippets only, the extension could provide completions based on cursor position and file context.

Examples:

- record type completion after `record(`
- valid field names for a specific record type
- enum values for fields such as `SCAN`, `OMSL`, `SELM`, `DTYP`
- macro name completion inside `$(...)` or `${...}`
- file path completion inside `dbLoadRecords`, `dbLoadTemplate`, and `include`
- record link target completion for `INP`, `OUT`, `FLNK`, `SELL`, `DOL`

This requires the extension to know:

- what kind of file is being edited
- what syntactic construct the cursor is inside
- what schema applies at that location

### 2. Diagnostics

The extension could validate EPICS source and report problems in the Problems panel.

Examples:

- unknown record types
- invalid fields for a given record type
- invalid enum values
- duplicate record names
- unresolved record links
- unresolved macros
- missing template arguments
- invalid startup commands or missing files

This is one of the most important upgrades because it changes the extension from a text helper into a correctness tool.

### 3. Hover Documentation

Hover should show semantic information, not just token coloring.

Examples:

- field description, type, and meaning for `SCAN`, `DTYP`, `INP`, `VAL`
- enum documentation for fields with constrained values
- record type documentation
- macro source and resolved value when available
- file and expansion information for loaded templates

The natural source of truth here is `.dbd` data plus workspace indexing.

### 4. Go To Definition

Navigation is a major feature that makes a language feel real.

Examples:

- jump from a record link to the referenced record
- jump from `dbLoadRecords("foo.db")` to `foo.db`
- jump from `dbLoadTemplate("foo.sub")` to `foo.sub`
- jump from substitutions entries to the target template file
- jump from macro usage to its defining `epicsEnvSet` or template parameter site

### 5. Find References

Once the workspace is indexed, the extension can answer cross-reference questions.

Examples:

- where a record is used
- where a macro is referenced
- where a template is instantiated
- where a startup file loads a database
- what records point to a given PV or record name

### 6. Rename Symbol

Rename is difficult to do safely with plain text search, but practical with a symbol graph.

Good rename targets include:

- record names
- macro names
- template parameters
- optionally PV prefixes where usage is structured enough

### 7. Document Symbols and Outline

The extension could populate the VS Code outline/tree with structured items.

Examples:

- records in `.db` files
- fields under each record
- `recordtype`, `field`, `menu`, and `choice` blocks in `.dbd`
- file and pattern blocks in substitutions files
- IOC shell commands in startup files
- protocol blocks in `.proto`

### 8. Formatting

A formatter could normalize style across EPICS files.

Examples:

- indentation in records and blocks
- field alignment
- spacing around commas and parentheses
- substitutions table alignment
- startup command formatting

This is a strong quality-of-life feature and easier to adopt when backed by a parser.

### 9. Code Actions

Once diagnostics exist, the extension can offer automatic fixes.

Examples:

- add missing common fields to a record skeleton
- replace invalid enum text with a valid choice
- create a missing template argument
- convert a plain string into a macro reference
- generate a record body from a record type

### 10. Semantic Highlighting

TextMate highlighting is regex-driven. Semantic highlighting can color tokens according to meaning.

Examples:

- known record identifiers vs plain strings
- unresolved record links
- field names recognized through loaded `.dbd`
- custom module record types and fields
- macro definitions vs macro uses

Semantic highlighting does not replace the TextMate grammar. It augments it.

## EPICS-Specific Features With High Value

The following features are especially important for EPICS and would distinguish this extension from generic text tooling.

### DBD-Driven Intelligence

The `.dbd` files define record schemas and metadata. They should become the semantic backbone of the tool.

Use cases:

- validate whether a field belongs to a record type
- show field documentation in hover
- provide only valid field completions
- recognize custom record types and fields from site-specific modules

Without `.dbd` integration, the tool will stay limited and brittle.

### Workspace Indexing

The extension should understand the EPICS workspace as a graph, not isolated files.

Relevant file types include:

- `.db`
- `.template`
- `.dbd`
- `.sub`, `.subs`, `.substitutions`
- `.cmd`, `.iocsh`
- `.proto`
- `.st`

The index should capture:

- records
- record types
- fields
- macros
- includes
- load operations
- template instantiations
- link references

### Macro Resolution

EPICS projects often rely heavily on macro expansion.

The extension should track:

- `$(MACRO)`
- `${MACRO}`
- macro defaults
- `epicsEnvSet(...)`
- substitution pattern/value expansion
- startup-time parameter propagation

Even partial macro resolution would be valuable if it can:

- display candidate values
- flag unresolved references
- explain the resolution chain

### Template Expansion Preview

This is one of the most useful EPICS-specific features.

For `.template` plus substitutions, the extension could:

- show expanded records in a preview panel
- show per-row expansion output
- validate whether all required macros are provided
- show resulting record names and collisions

This would save developers from mentally simulating substitutions.

### Link Analysis

This enables:

- navigation

Important commands include:

- `dbLoadRecords`

This would let the tool answer:

- which databases are loaded by this IOC

### StreamDevice and Related Protocol Support

The extension already highlights `.proto` files lexically. A stronger version would connect them semantically to record usage.

## Architectural Direction

To support the above features, the extension should move toward a semantic architecture.

Recommended high-level structure:

1. Parsers for EPICS file types
2. Shared semantic model / workspace index
3. Language server implementing editor features
4. Thin VS Code client extension

### 1. Parsers

Each EPICS DSL should be parsed into an abstract syntax tree or equivalent structured representation.

Priority parsers:

- `.db`
- `.dbd`
- `.sub`
- `.cmd` / `.iocsh`

Secondary parsers:

- `.proto`
- `.st`

The parser does not need to be overly academic at the start. It does need to be reliable enough to recover from incomplete user input during editing.

Requirements:

- incremental or fast reparsing
- error recovery
- source ranges for all nodes
- enough structure for diagnostics and navigation

### 2. Semantic Model / Workspace Index

The parser output should feed a shared index.

Possible indexed entities:

- record declarations
- record types
- fields
- macros
- macro assignments
- template parameters
- substitutions rows
- startup load edges
- file includes
- links between records

This is the core of the future system. Most editor features become queries over this graph.

### 3. Language Server

A Language Server Protocol (LSP) implementation is the standard way to expose language intelligence in VS Code.

The server would provide:

- completion
- hover
- diagnostics
- go to definition
- references
- rename
- document symbols
- semantic tokens
- formatting
- code actions

Benefits of using LSP:

- clean separation from editor UI concerns
- easier testing
- potential reuse outside VS Code later
- better long-term architecture than adding ad hoc commands in the extension host

### 4. Thin VS Code Client

The VS Code extension itself should stay thin.

Responsibilities:

- start the language server
- wire editor commands and views
- keep current snippet and TextMate contributions
- provide any custom preview UI such as template expansion or graph visualization

This avoids bloating the current manifest-centric extension into a large monolith.

## TextMate vs Semantic Tooling

The current extension should not discard its existing grammars and snippets.

Instead:

- keep TextMate grammars for baseline syntax coloring
- keep snippets for fast text insertion
- add semantic tooling on top through the language server

This layered design is standard and practical.

TextMate remains good for:

- initial highlighting
- low-cost token styling
- bootstrapping support quickly

Semantic tooling is needed for:

- correctness
- structure-aware editing
- cross-file understanding
- refactoring

## Practical Development Strategy

A pragmatic path would be:

1. Keep the existing extension contributions as they are.
2. Add a small runtime extension client.
3. Build a language server beside the current grammar/snippet assets.
4. Introduce parsing and indexing incrementally.
5. Reuse `.dbd` as the schema source whenever possible.

This allows a gradual migration:

- existing users keep syntax highlighting and snippets
- advanced features can land without breaking the current extension

## Risks and Challenges

Several challenges should be expected.

### Incomplete or Site-Specific EPICS Schemas

Different facilities may use custom modules and custom `.dbd` files.

Implication:

- the tool must be able to load external schemas
- hardcoding base EPICS symbols is not enough

### Macro Complexity

Macro expansion can be dynamic and context-sensitive.

Implication:

- resolution may need to be partial rather than absolute
- the UI should explain uncertainty rather than pretend to know

### Error Recovery During Editing

Users edit incomplete files.

Implication:

- parsers must tolerate missing delimiters, broken strings, and partial records
- diagnostics should remain stable while typing

### Cross-File Performance

Workspace indexing can become expensive.

Implication:

- cache parse results
- update incrementally
- avoid full re-indexing on every keystroke

## What Success Looks Like

The extension should eventually support workflows like these:

- Type `field(` and get only valid fields for the current record type.
- Hover a field and see schema docs from `.dbd`.
- Click a record link and jump to its definition.
- Open a startup file and navigate into every loaded database and template.
- Open a substitutions file and preview generated records.
- Rename a record safely across the workspace.
- See errors for invalid fields, missing macros, unresolved links, and duplicate records before runtime.

At that point, EPICS support in VS Code would feel much closer to support for a real language ecosystem rather than an editor-only syntax package.

## Standard EPICS Application Structure

This section captures recommendations for supporting a common EPICS application layout based on the example application at `/Users/1h7/projects/vscode-epics/appTest`.

### Observed Layout

The sample application follows the standard EPICS application skeleton:

- `configure/`
- `iocBoot/`
- `<name>App/`

In the concrete example:

- `configure/RELEASE` defines external support and `EPICS_BASE`
- `iocBoot/ioctest01/st.cmd` is the IOC startup script
- `test01App/src/Makefile` defines `PROD_IOC = test01` and `DBD += test01.dbd`
- `test01App/Db/Makefile` is the source database installation point
- `test01App/src/test01Main.cpp` is the IOC main entry point

This structure is important because EPICS developers usually work in source-tree paths while startup scripts often reference generated runtime paths such as `../../db/...` and `../../dbd/...`.

### Why This Matters for the Extension

For a standard EPICS app, the extension should understand both:

- the source layout
- the build/install layout implied by Makefiles

Without that mapping, startup assistance remains shallow. For example:

- `dbLoadDatabase "../../dbd/test01.dbd"` in `st.cmd` refers to a generated runtime artifact
- the source of truth for that artifact is `test01App/src/Makefile`
- `dbLoadRecords("../../db/test01.db", ...)` refers to an installed database under `db/`
- the source of truth for that file is usually `*App/Db/Makefile` plus the corresponding `.db` source file

### Recommended Extension Features for This Structure

#### 1. EPICS App Root Detection

The extension should automatically detect an EPICS application root when it finds a layout like:

- `configure/RELEASE`
- `iocBoot/`
- one or more `*App/` directories

Once detected, the extension can switch into a more opinionated EPICS mode without requiring manual configuration.

#### 2. Makefile-Aware Build Graph

The extension should parse standard EPICS Makefiles to infer generated outputs and relationships.

High-value symbols to extract:

- `PROD_IOC`
- `DBD +=`
- `<ioc>_DBD +=`
- `DB +=`
- `DIRS +=`

This enables the extension to infer:

- IOC executable name
- generated `.dbd` files
- installed DB/template files
- likely source-to-runtime file mapping

For the sample app:

- `PROD_IOC = test01` implies the IOC binary name
- `DBD += test01.dbd` implies a generated runtime DBD
- `test01_DBD += base.dbd` contributes to the generated DBD

#### 3. Startup Script Intelligence

The extension should treat `iocBoot/*/st.cmd` as a semantic hub.

For `st.cmd`, it should understand:

- `dbLoadDatabase`
- `dbLoadRecords`
- `dbLoadTemplate`
- `epicsEnvSet`
- `iocInit`
- generated `*_registerRecordDeviceDriver(...)` calls

The provider should resolve these against both:

- actual files in the source tree
- inferred build products from Makefiles

#### 4. Source-to-Runtime Path Resolution

This is one of the most useful structure-aware features.

Examples:

- `../../dbd/test01.dbd` in `st.cmd` should navigate to the relevant DBD definition source or generating Makefile entry
- `../../db/foo.db` should resolve to the matching source database under `*App/Db/`

The extension should present a logical source location to the user, not only the installed runtime path.

#### 5. Diagnostics for Build/Startup Mismatches

Once the build graph is understood, the extension can report common structural errors.

Examples:

- startup script loads a DB that is not declared in `DB +=`
- startup script references a `.dbd` that is not generated by the app
- `*_registerRecordDeviceDriver` name does not match the IOC or DBD product
- source DB exists but is not installed
- installed DB declared in `Makefile` has no backing source file

For the sample app, this is already relevant because:

- `test01App/Db/Makefile` currently has no `DB += ...` entries
- `iocBoot/ioctest01/st.cmd` contains a commented-out `dbLoadRecords("../../db/test01.db", ...)`

That is exactly the kind of cross-file consistency check the extension should surface.

#### 6. RELEASE-Aware External Module Resolution

`configure/RELEASE` should be parsed so the extension can discover external support trees.

This matters for:

- external `.dbd` schema loading
- support module DB/template lookup
- future C/C++/SNL assistance tied to module paths

In practice, this allows custom site modules to participate in completion, hover, diagnostics, and navigation.

#### 7. IOC-Oriented Commands and Tasks

For a standard app layout, the extension could offer commands or generated tasks such as:

- build the application
- rebuild after `RELEASE` changes
- run a selected IOC from `iocBoot/<ioc>/`
- open the active `st.cmd`
- validate startup references

This should be driven by the app structure rather than hardcoded per project.

### Suggested Semantic Model Additions

To support the above structure, the semantic layer should include EPICS app-level entities in addition to language-level symbols.

Suggested entities:

- `EpicsApplication`
- `EpicsIocBoot`
- `EpicsIocBinary`
- `EpicsMakefileTarget`
- `InstalledDatabaseFile`
- `InstalledDbdFile`
- `ReleaseVariable`

Suggested relationships:

- application contains app module
- app module builds IOC
- IOC uses startup script
- startup script loads installed DB/DBD
- installed DB/DBD maps back to source declaration
- `RELEASE` variable resolves external module root

### Best Next Steps for This Structure

The most valuable near-term additions for a standard EPICS application layout are:

1. detect EPICS app roots automatically
2. parse Makefiles for `PROD_IOC`, `DBD +=`, and `DB +=`
3. resolve `st.cmd` load paths against inferred runtime outputs
4. add diagnostics for missing DB/DBD declarations and broken startup references
5. support navigation from runtime startup paths back to source files

### Summary for `appTest`

For the example application, the extension should not treat files like `st.cmd`, `Db/Makefile`, `src/Makefile`, and `configure/RELEASE` as unrelated text files. They form a standard EPICS application graph.

If the extension models that graph, it can provide:

- better path completion in startup files
- more accurate navigation
- diagnostics for broken build/startup integration
- awareness of generated runtime artifacts
- support for external modules defined in `RELEASE`

This is a high-value direction because the structure is common across EPICS applications and can be supported with little or no manual project configuration.
